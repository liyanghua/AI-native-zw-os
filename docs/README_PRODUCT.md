# AI-native 经营操盘系统

## 1. 产品定义

这是一个围绕商品经营主线运转的 AI-native 经营操作系统。

**与本仓库对齐的可运行界面原型**：Vite + React Router 单页应用；侧栏导航、角色切换与 URL 路径以 `src/app/routes.tsx` 与 `docs/IA_AND_PAGES.md` 为准（界面标题区展示「AI-native · 经营操盘系统V2」）。领域对象与字段定义仍以 `docs/DATA_MODEL.md` 为权威来源。

它不是传统 BI 看板，不是普通项目管理工具，也不是单个聊天助手前台。
它的目标是把企业围绕商品经营的主线——从商机发现、开新品、首发验证、增长优化、老品升级，到复盘沉淀——统一到一个系统里。

### 当前实现阶段：Local Pilot Sandbox / Batch 3

当前仓库已经从“页面级 mock 原型”升级为**本地单机闭环沙箱**的第三批实现：

- 本地 SQLite 数据底座已落地，数据库文件默认位于 `data/pilot-sandbox.sqlite`
- 已实现本地 Node API：
  - `GET /api/projects`
  - `GET /api/projects/:id`
  - `GET /api/projects/:id/knowledge`
  - `POST /api/knowledge/search`
  - `POST /api/brain/compile-context`
  - `POST /api/brain/compile-decision`
  - `POST /api/brain/compile-role-story`
  - `GET /api/roles/:role/dashboard`
- 已 seed 3 个演示项目，用于演示首发验证、增长优化、复盘沉淀
- 已 seed 本地知识资产与 chunk，用于 SOP / rule / case / template / evaluation sample 检索
- 生命周期总览 / 生命周期阶段页 / 项目详情页已经切换为 `SQLite -> API -> repository -> UI`
- 项目详情页已经升级为“项目 + 证据 + 决策”页面，可展示：
  - fact evidence
  - method evidence
  - 结构化 `DecisionObject`
  - `boss` / `operations_director` / `product_rnd_director` / `visual_director` 的同源 `RoleStory`
- 角色首页已从泛化 dashboard 升级为基于角色原型的同源闭环入口：
  - `boss` 已正式迁移
  - `operations_director` 已正式迁移
  - `product_rnd_director` / `visual_director` 已有 API-backed 框架页
- 其他页面仍保留现有 `pilotRuntime` / repository-first 原型能力，作为后续批次迁移对象

当前**尚未实现**：

- 向量数据库 / embedding pipeline
- Agent trigger
- 执行 writeback
- review generate
- asset publish
- 完整动作中心与审批流
- 更完整的 product_rnd_director / visual_director 角色深度编排

也就是说，Batch 3 的目标是把**角色闭环入口**建立起来，而不是假装执行、回写和资产闭环已经全量接通。

系统中的职责分工如下：

- 人类管理者负责：方向、取舍、审批、学习
- 经营大脑负责：目标理解、上下文编译、决策生成、归因分析、资产调用与沉淀建议
- 场景 Agent 负责：沿商品生命周期推进项目，生成中间结果，推动状态流转
- 执行与自动化层负责：执行动作、更新系统、回写结果、触发告警
- 实时更新层负责：让经营信号、项目状态、风险、待批动作、Agent 状态持续更新

## 2. 为什么这样设计

传统经营系统的主要问题：

### 2.1 按部门和功能切分，而不是按经营主线切分
商机、研发、视觉、投放、复盘分散在不同系统里，信息断裂、目标断裂、协作断裂。

### 2.2 管理层仍被迫处理大量执行细节
当执行能力可以由 Agent 和 runtime 承担时，管理层应上移到目标、资源、审批、例外、组织学习。

### 2.3 看到了问题，但无法形成闭环
很多系统只能做监控和分析，无法把问题转成决策对象、动作、执行、结果、经验和资产。

因此本系统采用：
- 生命周期主轴
- 项目对象中心
- 角色视图
- 人 × 大脑 × 场景 Agent × 执行自动化协同

## 3. 核心设计原则

### 3.1 生命周期做主骨架
- 机会池
- 新品孵化
- 首发验证
- 增长优化
- 升级迭代
- 复盘沉淀

### 3.2 项目对象做协作中心
所有角色、场景、动作都围绕 Project Object 协作，而不是围绕零散任务和功能模块协作。

### 3.3 角色只是默认视图，不是结构
老板、产品研发总监、运营&营销总监、视觉总监看到的是同一套底层对象和状态，只是默认入口、优先级和摘要不同。

### 3.4 人只做高价值判断
人负责方向、取舍、高风险审批、低置信度判断、标准化确认。

### 3.5 大脑负责编译，不替代人拍板
大脑的职责是理解、编译、建议、归因，不是代替管理者做关键取舍。

### 3.6 Agent 负责推进，自动化负责跑通
Agent 更偏流程推进和中间产物生成；自动化更偏执行、回写、告警和触发。

### 3.7 实时更新只放在状态、信号和风险上
商品定义、视觉策略、复盘结论等是阶段性产物，不要求实时刷新。

## 5. 主线故事

整个系统围绕商品项目运转，主线为：

1. 发现机会
2. 立项与定义
3. 表达与首发准备
4. 首发验证
5. 增长优化
6. 升级迭代
7. 复盘沉淀

人、大脑、Agent、自动化都围绕这条主线协同。

## 6. 当前阶段的产品验证逻辑

当前阶段不是继续无限扩页面，而是要围绕三个阶段逐步验证产品是否成立：

- 阶段 A：先验证“主线是否成立”
- 阶段 B：再验证“人 + 大脑 + Agent + 执行”的协同是否清楚
- 阶段 C：最后验证“闭环是否成立”

这三个阶段不是独立项目，而是同一产品逐步验证的三层递进关系：

### A 先验证“主线是否成立”
先证明这套产品不是“很多页面的集合”，而是“围绕商品经营主线推进的一套操盘系统”。

### B 再验证“协同是否清楚”
在主线成立的前提下，证明用户能看懂：
- 人在做什么
- 大脑在做什么
- Agent 在做什么
- 执行端在做什么

### C 最后验证“闭环是否成立”
在主线和协同都成立的前提下，证明系统不是停在建议层，而是真的能从问题走到动作、执行、回写、复盘、资产沉淀。

---

## 7. 阶段 A：验证“主线是否成立”

### 7.1 阶段目标
验证系统是否真的按商品经营主线组织，而不是只是把一堆功能页面放在一个壳子里。

### 7.2 要验证的核心假设
1. 生命周期主轴是否比按角色、按模块导航更符合管理者理解
2. 用户是否愿意沿“商机 → 新品 → 首发 → 增长 → 复盘”的主线推进
3. 商品项目详情页是否能成为跨角色协作中心

### 7.3 对应的产品设计逻辑
这个阶段的重点不是做更多页面，而是让产品结构足够清楚：

- 顶层用“商品经营主线”作为主导航
- 场景作为阶段下的工作站存在
- 所有关键协作最终落到“商品项目详情页”
- 老板与三位总监看到的是同一条主线，只是入口和摘要不同

也就是说，这个阶段的产品设计核心是：

**让系统先成为一条主线，而不是一堆页面。**

### 7.4 对应的交互逻辑
用户进入系统后，应该能够自然形成下面的交互路径：

1. 先看到“今天最值得关注的经营脉冲”
2. 再进入“商品经营主线总览”
3. 再进入某个生命周期阶段
4. 最后落到具体“商品项目详情页”

因此这个阶段的交互重点是：

- 主线总览必须清晰
- 阶段切换必须自然
- 项目详情页必须成为主线汇合点
- 不同角色进入后，能围绕同一个项目页展开讨论

### 7.5 本阶段优先页面
- 经营指挥台
- 商品经营主线总览
- 商机池
- 新品孵化
- 首发验证
- 商品项目详情

### 7.6 本阶段成功标准
如果老板和三位总监都能快速回答下面这些问题，说明主线成立：

- 这个项目现在在哪个阶段？
- 当前最大问题是什么？
- 下一步是什么？
- 谁需要拍板？

---

## 8. 阶段 B：验证“人 + 大脑 + Agent + 执行”的协同是否清楚

### 8.1 阶段目标
验证这套系统中的 AI-native / Agentic 分工是否能被管理者理解和接受。

### 8.2 要验证的核心假设
1. 用户是否能区分“大脑建议”和“Agent 推进”
2. 用户是否能接受“常规动作自动推进，关键动作人工审批”
3. 用户是否信任大脑给出的判断和证据

### 8.3 对应的产品设计逻辑
这个阶段的重点不是加更多 AI，而是让协同分工可见：

- 大脑负责判断、编译、建议、归因
- 场景 Agent 负责推进阶段动作和中间产物
- 执行端负责真正执行、回写、触发
- 人负责方向、审批、接管、学习

因此在产品设计上，必须避免“AI 像一个模糊助手”的表达，而要明确表现为：

- 大脑建议区
- Agent 状态区
- 执行状态区
- 人工节点区

也就是说，这个阶段的产品设计核心是：

**让分工看得见，而不是让 AI 变多。**

### 8.4 对应的交互逻辑
在关键页面中，用户应该一眼看清：

- 当前建议是谁给出的
- 当前推进是谁在做
- 当前动作是否已经执行
- 当前是否在等人工拍板

因此这个阶段的交互重点是：

- 增加协同状态条
- 增加来源标记（来自大脑 / Agent / 自动执行 / 人工）
- 增加置信度与证据入口
- 增加待人工处理的明确提示

### 8.5 本阶段优先页面
- 商品项目详情
- 运营与营销总监工作台
- 产品研发总监工作台
- 视觉总监工作台
- 动作中心
- 风险与审批台

### 8.6 本阶段成功标准
如果用户能清楚复述下面这些问题，说明协同成立：

- 这个建议是谁给的？
- 这一步是谁在推进？
- 哪些动作已经自动执行？
- 哪一步还需要我来决定？

---

## 9. 阶段 C：验证“闭环是否成立”

### 9.1 阶段目标
验证系统能否真正从“看问题”走到“形成闭环”，而不是停在建议层。

### 9.2 要验证的核心假设
1. 建议是否能自然转成动作
2. 动作是否能看见执行与回写
3. 结果是否能进入复盘
4. 复盘是否能沉淀成资产
5. 资产是否能为下一轮提供复用价值

### 9.3 对应的产品设计逻辑
这个阶段的重点不是再加分析页，而是把链路打通：

- 问题进入决策对象
- 决策对象进入动作生命周期
- 动作被执行端推进并回写
- 结果进入复盘摘要
- 复盘进入资产候选
- 资产进入经验资产库，供下一轮调用

也就是说，这个阶段的产品设计核心是：

**从“建议系统”升级成“闭环系统”。**

### 9.4 对应的交互逻辑
用户应该能够沿着一条连续链路操作，而不是跳散页面：

1. 在项目页看到问题和建议
2. 进入动作中心处理动作
3. 在风险与审批台处理高风险项
4. 回到项目页看到结果回写
5. 进入复盘沉淀页看归因和经验
6. 在经验资产库看到已沉淀内容

因此这个阶段的交互重点是：

- 页面之间有明确主线联动
- 项目页与动作中心、复盘沉淀之间关联清楚
- 状态流要能看见“建议 → 动作 → 执行 → 回写 → 复盘 → 资产”
- 不要求全部自动化，但必须能表达闭环

### 9.5 本阶段优先页面
- 商品项目详情
- 动作中心
- 风险与审批台
- 复盘沉淀
- 经验资产库

### 9.6 本阶段成功标准
如果用户能够完整走通一条最短链路，说明闭环成立：

- 看见问题
- 看到建议
- 形成动作
- 执行状态可见
- 结果回写可见
- 复盘与资产候选可见

---

## 10. 当前阶段最小验证故事线

当前推荐的最小验证故事线是：

**商机 → 新品孵化 → 首发验证 → 放量/调整 → 复盘沉淀**

为什么选这条线：

1. 生命周期最完整
2. 涉及产品研发总监、运营&营销总监、视觉总监
3. 老板可以在关键节点做拍板
4. 能同时验证主线、协同与闭环
5. 是最适合现阶段原型走查的一条经营主链路

## 11. 当前阶段优先级

### 第一优先级
- 把商品项目详情页补成主线协作中心
- 让生命周期主轴更容易走通
- 让老板和三位总监都能围绕同一个项目页讨论

### 第二优先级
- 让“大脑 / Agent / 执行 / 人”的分工可见
- 让待决策、待审批、已执行、已回写状态清楚

### 第三优先级
- 让动作中心、风险与审批台、复盘沉淀、经验资产库形成最小闭环

---

## 12. 当前阶段不优先做的事情

当前阶段不优先：

- 扩更多不相关页面
- 接入真实后端
- 做完整权限系统
- 做复杂多租户逻辑
- 做真正的多 Agent 编排后端
- 做精细化视觉打磨超过验证所需

当前阶段优先做的是：

**把最短主线走通，把协同看清，把闭环表达出来。**

---

## 13. 非目标

本项目当前不做以下内容：

- 不做复杂多租户权限系统的完整实现
- 不做真实大模型/真实 Agent 运行时集成的完整后端，只保留对接位和 mock state
- 不做所有生命周期阶段的真实业务策略，只先把前台框架、状态流和对象模型搭起来
- 不做最终设计系统和品牌视觉定稿，只先做可用且一致的原型级 UI
- 不追求“页面齐全”，优先追求“主线、协同、闭环成立”
